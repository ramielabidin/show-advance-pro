Scope selector — mobile elegance study

The All Shows page currently packs a segmented Upcoming/Past control alongside three origin pills (All / Tour ▾ / Standalone). On a 390 px phone they wrap onto two lines and read as one flat group of five chips of equal weight. Time scope and origin scope deserve different visual hierarchy.

This study compares three directions for a calmer mobile row, with each direction rendered in three states at mobile width and one canonical state at desktop. All chrome above and below the selector is real Advance UI, so the selector reads in context.

Direction A

Two-line hierarchy

Promote time scope to the primary control: a full-width segmented Upcoming/Past on its own row. Origin scope drops to a quieter second row of left-aligned pills. The two scopes read as different things, so the eye can scan them separately. Costs vertical space — one extra row on mobile.

Direction B

Single row, origin behind a popover

Keep one row. Upcoming/Past on the left, a single labelled trigger button on the right collapses origin into a popover (All shows / On tour: … / Standalone). The trigger always shows the current selection, so origin is one tap away without a second row. Costs a tiny bit of discoverability — the user has to learn the popover.

Direction C

Filter chip — origin only when set

Default state shows just the Upcoming/Past segmented control plus a small funnel icon. Origin is implied "All". The moment a tour or Standalone is selected, an active filter chip appears with × to clear. Quietest default — most sessions never see origin chrome — and the active filter feels like a deliberate state, not a permanent control.

Recommendation

Direction C · Filter chip when set, nothing when not

For most sessions on this page the user is browsing "all shows", and showing three idle origin pills as permanent chrome trains the eye to ignore them. Direction C makes the page quieter by default and more deliberate when a filter is applied — the chip reads like a state, not a control.

It also collapses cleanly across breakpoints: mobile and desktop use the same row pattern, just with the chip on a wider canvas. No second-row hierarchy to maintain (Direction A) and no popover convention to teach (Direction B).

Trade-offs to talk through before locking in:

  • Discoverability of "filter by tour". The funnel icon is the only invitation in default state. If "filter to a specific tour" is a primary use case (e.g. tour managers usually live inside one tour), Direction A or B may be safer because the affordance is always labelled.
  • Chip width on long tour names. Truncating to ~16 chars on mobile is necessary; the popover preserves the full name. Worth confirming common tour-name length in real data.
  • "Standalone" as a filter chip is slightly weird — it's not a tour-based filter, it's a content-type filter. Direction B's labelled trigger reads more naturally for that case.
